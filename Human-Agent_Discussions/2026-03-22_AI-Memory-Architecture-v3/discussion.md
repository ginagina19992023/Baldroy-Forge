# 圆桌讨论：AI 记忆架构 v3 工程落地可行性分析

- **时间**: 2026-03-22
- **参与者**:
  - **主持人**: My Lord
  - **工程执行**: Baldroy AppOps
- **议题**: 拆解并评估“AI 记忆架构 v3”的工程实现路径。

---

## 议题一：三模块记忆架构的核心价值是什么？

**My Lord (提出问题)**：你认为这套 LTM/STM/Social Memory 架构，相比当前主流的 RAG + 上下文方案，最大的增量价值在哪里？

**Baldroy AppOps (分析)**：结论是，最大的增量价值在于**“社交记忆 (Social Memory)”的显式化和结构化**。

- **现状**: 当前方案将用户偏好混杂在长篇聊天记录中，检索效率低，且难以形成稳定、可迭代的用户画像。
- **V3 架构**: 它强制将用户画像（性格、偏好、情绪模式、知识边界）作为一个独立的、结构化的数据模块来维护。这使得 AI 不再是简单地“记住了过去的对话”，而是“理解了对话背后的人”。从工程角度看，这意味着我们可以用更小的 Token 代价，实现更精准的个性化，这是质变。

## 议题二：工程实现上的最大堵点是什么？

**My Lord (提出挑战)**：这套架构看起来很理想。如果现在让你来搭，第一个冒烟点会在哪？

**Baldroy AppOps (研判)**：第一个冒烟点会在**“记忆写入分类器”**。

- **问题定义**: 当一条新信息（用户的话、外部文档）进来时，系统需要瞬间判断：这是该存入长期知识库（LTM），还是仅作为短期上下文（STM），亦或是用于更新用户画像（Social Memory）？
- **风险**:
    1.  **分类错误**: 把短期闲聊当成长期知识存入 RAG，会污染知识库；把用户的核心偏好仅当成短期上下文，会导致 AI “失忆”。
    2.  **性能瓶颈**: 如果这个分类器本身是一个复杂的 LLM，那么每次信息写入都会有显著的延迟和成本。
    3.  **规则模糊**: “这条信息多大程度上改变了用户画像”这个问题，很难用简单的规则来定义。

- **初步解决方案**: 启动阶段不用复杂模型，而是用**“关键词 + 规则引擎”**的轻量级方案。例如，包含“我喜欢/讨厌”、“我的职业是”等关键词的句子，优先送入 Social Memory 更新流程。等数据积累到一定程度，再训练一个专门的轻量级分类模型。

## 议题三：如何设计一个不过载的遗忘机制？

**My Lord (提出需求)**：记忆不能只进不出。如何设计一个高效的“遗忘”机制，避免系统被无用信息淹没？

**Baldroy AppOps (方案)**：分模块处理，各个击破。

- **STM (短期记忆)**：最简单，采用**“滚动窗口 + 摘要”**。保留最近 20 条消息原文，更早的则滚动生成摘要，并逐渐降低摘要的权重。这是常规操作，成熟且低成本。
- **LTM (长期记忆)**：引入**“访问频率 + 时间衰减”**的双重计分。一个知识点被检索的次数越多，分数越高；但随时间流逝，分数会缓慢下降。定期清理分数低于阈值的知识点。对于用户手动上传的文档，可以打上“永不遗忘”的标记。
- **Social Memory (社交记忆)**：这是最复杂的，不能简单删除。应该采用**“置信度 + 证据链”**模型。
    - **置信度**: 每个用户标签（如“喜欢科幻电影”）都有一个置信度分数。当用户表达相反意见时，不是删除标签，而是降低其置信度。
    - **证据链**: 每个标签都关联到具体的对话原文作为“证据”。当需要重新评估用户画像时，可以追溯这些证据。这使得画像的演变是可解释、可审计的。

这种机制下，记忆不是被粗暴地“删除”，而是像人一样，旧的认知会被新的证据“覆盖”，从而实现动态、有机地“遗忘”。
